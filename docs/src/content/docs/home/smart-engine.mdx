**Ryanlink** is powered by a proactive "Smart Layer" designed to handle the complexities of Discord audio automatically. This architecture ensures that your users experience zero-jitter playback and consistent sessions, even during node failures or bot restarts.

## Technical Overview

### Session Persistence & Resuming

Ryanlink uses high-resolution timestamp tracking to ensure that when a player is recreated, it starts at the exact millisecond where it left off.

<Steps>
1. **Shadow Listeners**: A Proxy-based system monitors your queue for any changes.
2. **Auto-Save**: On any mutation (`push`, `shift`, `shuffle`), the state is pushed to the `QueueStore`.
3. **Session Verification**: Upon instantiation, the Player checks for existing saved states.
4. **Millisecond Sync**: The `play()` command is issued with the precise `startTime`.
</Steps>

### High-Availability Failover

Load balancing is not just about player counts; it's about node health.

- **Weighted Scoring**: Nodes are scored based on CPU usage, memory availability, and frame deficits.
- **Failover Logic**: If a node disconnects, the Manager automatically selects the node with the lowest weighted score and migrates all active players atomically.

## Lifecycle Hooks <Badge text="New in 2.0" variant="tip" />

You can now respond to engine-level events directly within the player options:

```typescript
const player = manager.createPlayer({
  guildId: '...',
  voiceChannelId: '...',
  // Triggered when a track starts
  onTrackStart: (p, track) => { ... },
  // Triggered when the queue is naturally exhausted
  onQueueEnd: (p) => { ... },
  // Triggered after a successful node migration
  onNodeFailover: (p, oldNode, targetNode) => { ... }
});
```
