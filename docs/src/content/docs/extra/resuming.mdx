Ryanlink supports two layers of persistence: **Lavalink session resuming** (reconnects the WebSocket with the same session ID so audio continues) and **queue persistence** (stores queue state so it survives bot restarts).

## Lavalink Session Resuming

Enable session resuming in your manager config:

```typescript
const manager = new RyanlinkManager({
  resuming: {
    enabled: true,
    timeout: 60000, // 60 seconds — Lavalink holds the session this long
  },
});
```

When the node reconnects, ryanlink automatically calls `node.updateSession(true, 60)`. On the `resumed` event, all player states are restored from Lavalink's server-side cache.

```typescript
manager.nodeManager.on('resumed', (node, payload, players) => {
  console.log(`Node ${node.id} resumed — ${players.length} players restored`);
  console.log(`Session ID: ${payload.sessionId}`);
});
```

### Manual Session ID

If you need to resume a specific session manually, pass `sessionId` in the node config:

```typescript
nodes: [{
  id: 'main',
  host: 'localhost',
  port: 2333,
  authorization: 'youshallnotpass',
  sessionId: 'your-previous-session-id',
}]
```

## Queue Persistence

Ryanlink uses a `QueueStore` to persist queue state. The default is in-memory (no persistence across restarts).

```typescript
import { RyanlinkManager, MemoryQueueStore } from 'ryanlink';

const manager = new RyanlinkManager({
  queueOptions: {
    queueStore: new MemoryQueueStore(), // default
    maxPreviousTracks: 25,
  }
});
```

### Custom Store

Implement the `QueueStore` interface — requires `get`, `set`, `delete`, `stringify`, `parse`:

```typescript
class RedisQueueStore {
  async get(guildId: string) {
    return redis.get(`queue:${guildId}`);
  }
  async set(guildId: string, data: any) {
    await redis.set(`queue:${guildId}`, this.stringify(data));
  }
  async delete(guildId: string) {
    await redis.del(`queue:${guildId}`);
  }
  stringify(data: any) { return JSON.stringify(data); }
  parse(data: string) { return JSON.parse(data); }
}

const manager = new RyanlinkManager({
  queueOptions: { queueStore: new RedisQueueStore() }
});
```

Ryanlink also ships `LocalDiskQueueStore` and `RedisQueueStore` built-in:

```typescript
import { MemoryQueueStore, LocalDiskQueueStore, RedisQueueStore } from 'ryanlink';

// Local disk (persists across restarts)
new LocalDiskQueueStore('./sessions');

// Redis (for sharded/clustered bots)
new RedisQueueStore(redisClient, 'mybot:queue:');
```

## Auto-Resume on Player Create

When `resuming.enabled: true`, creating a player automatically syncs its queue from the store and calls `player.autoResume()` to resume playback from the saved position.

```typescript
const player = manager.createPlayer({
  guildId: '...',
  voiceChannelId: '...',
  selfDeaf: true,
});
// Queue is restored automatically if a saved state exists
// player.autoResume() is called internally
```

## Previous Tracks Buffer

Ryanlink keeps a circular buffer of previously played tracks per player:

```typescript
queueOptions: {
  maxPreviousTracks: 25, // default
}

// Access in your commands
const prev = player.queue.previous; // Track[]
```

## Track Resolve Retries

For unresolved tracks (e.g. Spotify metadata resolved to YouTube), configure retry behavior:

```typescript
const manager = new RyanlinkManager({
  trackResolveRetryLimit: 3,      // global default
  autoSkipOnResolveError: true,   // skip to next track on failure
});

// Or per-player
manager.createPlayer({
  guildId: '...',
  trackResolveRetryLimit: 5,
});
```

When all retries are exhausted, `queueErrorReport` is emitted and the track is purged from the queue.
