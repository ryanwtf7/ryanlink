**Ryanlink** replaces traditional "manual resuming" with a proactive **Smart Persistence** layer. Instead of just resuming the WebSocket connection, the engine can now recover full player states, queues, and playback positions across entire process restarts.

## Automated Persistence

When `resume: true` is enabled in your `RyanlinkManager` configuration, the engine automatically manages state recovery without any additional code.

```typescript
const manager = new RyanlinkManager({
  // ... rest of your config ...
  resume: true,               // Enable automated session persistence
  resumeTimeout: 60,          // Retry window (seconds) for same-guild connections
  trackResolveRetryLimit: 3,  // Self-healing for unresolved tracks
});
```

### Supported Storage Drivers <Badge text="Scalable" variant="tip" />

Ryanlink supports multiple backend drivers to store your queue data.

| Driver | Best For | Requirement |
| :--- | :--- | :--- |
| **In-Memory** | Small bots | None (Default) |
| **Local Disk** | Medium/Large bots | Accessible filesystem |
| **Redis** | Clustered/Sharded bots | Redis server connection |

## Technical Workflow

### 1. Shadow Persistence
Whenever you interact with the player (`play()`, `pause()`, `add()`, `shuffle()`), the **Shadow Listener** Proxy detects the change and asynchronously persists the new state to your chosen `QueueStore`.

### 2. Zero-Config Recovery
If your bot process crashes or restarts:
1. Upon creating a new player (`manager.createPlayer()`), the engine checks the `QueueStore` for an existing session.
2. If found, it automatically **re-hydrates** the player state.
3. The track list is restored with its exact playback position.

### 3. Millisecond-Precise Resuming
When playback restarts, Ryanlink sends the exact `startTime` to the Lavalink node, ensuring that the music resumes from the precise millisecond where it was interrupted.

## Manual Resuming (Advanced)

If you need full control over the resumption process, you can manually provide a `sessionId` directly to your nodes:

```typescript
manager.nodes.add({
  host: '...',
  port: 2333,
  sessionId: 'your-previous-session-id', // Optional: Manual resuming
});
```

> [!TIP]
> For production bots, we strongly recommend using the **Redis** driver to ensure state is shared across multiple shards or containers.
